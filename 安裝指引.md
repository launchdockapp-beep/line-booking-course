# 安裝指引（可分段）— 帳號、設定、上傳

> 這是「人工設定」的單一清單，跟著 `lessons/` 一段一段做。每段只做該段、做完打勾、**STOP**。
> **上傳一律用 `gh`，徹底不碰 PAT / 明碼 token / `.env` / `deploy.sh`（已廢除）。**
> 最後更新：2026-06-23（已納入 agy 實測踩坑）

---

## A. 設定總覽（你會用到哪些服務、何時用）

| 服務 | 何時設定 | 用途 |
|---|---|---|
| GitHub ＋ `gh` CLI | 段 0（L0） | 放程式、跑 Pages、上傳（gh 授權，零 token） |
| Firebase（Firestore＋Auth） | 段 1（L1） | 資料庫＋交易＋登入 |
| LINE Login＋LIFF | 段 3（L3） | 會員自動認人 |
| LINE Messaging API（你的 OA） | 段 7A（L7A） | 推播（停課通知、到期提醒） |
| Cloudflare Worker＋Cron | 段 7A/7B | 排程與推播的伺服器 |

> 鐵律：**LINE Login channel 與 OA 的 Messaging API channel 必須在同一個 Provider**，userId 才一致。

## B. 分段設定（對應每一堂）

### 段 0 — 搭配 L0：GitHub ＋ gh ＋ Pages
1. 🧰 github.com → New repository → **Public** → 不勾任何初始檔 → Create。
2. 🧰 裝 `gh` 並一次性授權（**取代所有 PAT 做法**）：
   ```
   # macOS 沒有 gh 就先裝：
   brew install gh
   # 一次性瀏覽器授權（之後 push 永遠不用再碰 token）：
   gh auth login        # 選 GitHub.com → HTTPS → Login with a web browser
   ```
3. 終端機接上遠端並上傳：
   ```
   git init && git branch -M main
   git remote add origin https://github.com/<你>/<repo>.git
   git add . && git commit -m "L0" && git push -u origin main
   ```
4. 🧰 repo → Settings → Pages → Source `main` /(root) → Save。
- ✅ 段 0 完成：Pages 網址打得開；`git push` 不需輸入任何密碼/token。

### 段 1 — 搭配 L1：Firebase
1. 🧰 console.firebase.google.com（教室專用帳號）→ 建專案。
2. 🧰 Firestore → 建立 → 位置 `asia-east1` → **正式版模式**。
3. 🧰 Authentication → Sign-in method → 啟用 **匿名** ＋ **Google**。
4. 🧰 專案設定 → 新增 Web 應用 → 複製 `firebaseConfig`（只取值）。
5. 🧰 Authentication → Settings → 授權網域加 `你的帳號.github.io` 與 `localhost`。
6. 🧰 Firestore → 規則 → 貼 `安裝包/firestore.rules`，把 `STUDIO_EMAIL@gmail.com` 換成教室信箱 → **發布**。
- ✅ 段 1 完成：連線測試寫得進、讀得出。

### 段 2 — 搭配 L2：課表（無服務設定）
- 🧰 把 Canva 課表圖丟 Gemini 轉 CSV（欄位 `date,time,title,instructor,capacity`，看不清標 `???` 不准猜）。

### 段 3 — 搭配 L3：LINE Login ＋ LIFF
1. 🧰 developers.line.biz → 進 **OA 所在的同一 Provider** → Create channel → **LINE Login**。
2. 🧰 該 channel → LIFF → Add：Endpoint = Pages 網址、Size Full、scope `profile`+`openid` → 取得 **LIFF ID**。
3. 🧰 OA 圖文選單加一顆鈕 → 連結 `https://liff.line.me/你的LIFF_ID`。
- ✅ 段 3 完成：LINE 內點選單零輸入認人。

### 段 4–6 — L4/L5/L6：無新服務設定（純前端＋Firestore）。

### 段 7A — 搭配 L7A：Cloudflare Worker ＋ 停課通知
1. 🧰 取你 OA 的 **Channel access token**（Messaging API）。
2. 🧰 Cloudflare → Workers & Pages → Create → **Workers**（非 Pages）→ Deploy。
3. 🧰 Worker → Settings → Variables and Secrets（Type=Secret）：`LINE_TOKEN`、`PUSH_SECRET`（自訂隨機字）。
- ✅ 段 7A 完成：後台「停課並通知」推得出。

### 段 7B — 搭配 L7B：service account ＋ Cron
1. 🧰 Firebase 專案設定 → 服務帳戶 → 產生私密金鑰（JSON）。
2. 🧰 Worker Secret 加 `FIREBASE_SA`（整個 JSON 一行）、`PROJECT_ID`。
3. 🧰 Worker → Triggers → **Cron expression** 分頁填 `0 0 * * *`（UTC 00:00＝台灣 08:00）。
- ✅ 段 7B 完成：`/?run=1&secret=...` 測得到提醒與歸零。

## C. ⚠️ agy 實測踩坑清單（重跑前務必預先防範）

> 以下是上一輪 agy 真的踩過的坑（檢查 `line-booking-system` 成品得出）。每條都附「給 agy 的防呆指令」，**做對應功能前先把該條貼給 agy**。

### 坑 1（高風險／洩密）：把 `PUSH_SECRET` 放進會員端載入的檔
- **現象**：agy 把 `PUSH_SECRET`、`WORKER_URL` 寫在 `firebase-init.js`，而 `index.html`（會員端）也載入它 → 任何會員看原始碼就拿到推播密鑰，可冒用 Worker 亂發訊息、燒光免費額度。
- **防呆指令**：
  ```
  PUSH_SECRET 與 WORKER_URL 只能放在 admin.html（老師端），絕不可放進 firebase-init.js 或任何被 index.html 載入的檔。
  index.html 載入的 JS 一律不得含任何密鑰。
  ```

### 坑 2（race condition）：用 `setTimeout` 等登入就緒
- **現象**：`liff-init.js` 用 `setTimeout(initLIFF, 800)`、`checkMemberStatus` 每 500ms 重試，靠「猜延遲」等匿名登入/`db` 就緒——網路慢就壞、快就空等。
- **防呆指令**：
  ```
  不要用 setTimeout 等待 Firebase 或匿名登入就緒。改用 firebase.auth().onAuthStateChanged() 確認登入後再查 Firestore；
  或把匿名登入包成 Promise，initLIFF 內先 await 它。一律事件驅動，不要猜延遲。
  ```

### 坑 3（資料過時）：擅自開 Firestore 離線持久化
- **現象**：`firebase-init.js` 自行加了 `db.enablePersistence()`。預約系統最怕顯示**過時的名額/堂數**（會員看到舊 `enrolled`/`credits`），且多分頁會啟用失敗。安裝包沒要求這個。
- **防呆指令**：
  ```
  不要呼叫 db.enablePersistence()。預約類資料必須即時，名額與堂數要從伺服器讀最新值，不要用離線快取。
  ```
- 註：`runTransaction` 本身仍以伺服器為準、不會超賣；問題是「畫面顯示」會誤導。

### 坑 4（已廢路線復活）：又生出 PAT / `.env` / `deploy.sh`
- **現象**：agy 為了「CLI 自動化」生了讀 `.env` 的 `deploy.sh`，用 `export $(cat .env|xargs)`（值含空白會壞）、把 token 組進 push URL（會明文落地 `.git/config`）。
- **防呆指令**：
  ```
  上傳一律用 gh auth login（一次性瀏覽器授權）。不要產生 PAT、不要建 .env、不要寫 deploy.sh、不要把 token 組進 git remote URL。
  若 repo 內已有 deploy.sh / .env 範本，請刪除。
  ```

### 坑 5（格式不一致）：courseId 寫法前後不一
- **現象**：courseId 必須是去掉符號的 `YYYYMMDD_HHmm`（例 `20260625_1900`）。admin 寫入、前台查詢、`booking.js` 預約三處只要有一處格式不同就對不到課。
- **防呆指令**：
  ```
  courseId 一律用 date.replace(/-/g,'') + '_' + time.replace(/:/g,'')（即 YYYYMMDD_HHmm）。
  admin 寫入、前台查詢、預約三處必須完全一致，不可一邊用帶「-」「:」的格式。
  ```

### 坑 6（載入順序）：全域 `db` 依賴
- **現象**：`booking.js`、`liff-init.js` 依賴 `firebase-init.js` 的全域 `db`。script 順序錯或靠延遲等待就會 `db is undefined`。
- **防呆指令**：
  ```
  <script> 載入順序固定為：firebase SDK → firebase-init.js → liff-init.js → booking.js。
  不要用 setTimeout 補救順序；用事件/Promise 確保 db 就緒後才使用。
  ```

### 坑 7（跳過課前 gate）：直奔建檔、假設 gh 已登入
- **現象**：Flash 跑 L0 時直接生 manifest/sw/index.html，把「建 repo＋`gh auth login`」當背景略過，最後只用一句括號註解「browser-authorized gh CLI session」帶過，**沒叫學員登入、沒給指令、沒驗證**。沒授權過 gh 的學員一 push 就失敗。
- **根因**：agentic 模型會優先做「看起來像主任務」的事（建檔），課前人工 gate 不擋死就會被跳過。
- **防呆指令**：
  ```
  每堂課開始前，先把該課的 🧰 設定關卡原樣輸出給老師，並實際執行驗證指令（L0 是 `gh auth status`）。
  驗證未過就停下、給出明確指令（如 `gh auth login`）等老師完成，禁止假設、禁止只用括號註解帶過、禁止先建檔。
  ```
